# Tinker Training Usage Guide

This guide covers the core training protocols used in this repository with the Tinker framework. It includes both Reinforcement Learning (GRPO) and Supervised Fine-Tuning (SFT) with optimized asynchronous batching.

## 1. Reinforcement Learning (RL) Protocol

RL training in Tinker relies on **Trajectories**, which represent a sequence of agent interactions with the environment.

### Core Concepts

- **Transition**: The atomic unit (Observation + Action + Reward).
- **Trajectory**: A sequence of `Transition` objects.
- **TrajectoryGroup**: A collection of trajectories starting from the same prompt, used for relative advantage calculation (GRPO).

### Construction Protocol

1.  **Sampling**: Use `SamplingClient.sample_async` with `include_logprobs=True`.
2.  **Parsing**: Always use `Renderer.parse_response` to validate the message structure before construction.
3.  **Assembly**: Construct a `Trajectory` using the sampled tokens and assigned reward (binary `0.0/1.0` or scaled).

```python
from tinker_cookbook.rl.types import Transition, Trajectory, TrajectoryGroup

# 1. Parse and Verify
assistant_message, parse_success = renderer.parse_response(ac.tokens)
if not parse_success:
    reward = -1.0 # Or 0.0 with failure flag
else:
    reward = verify_execution(assistant_message)

# 2. Build Trajectory
transition = Transition(ob=prompt, ac=ac, reward=reward, episode_done=True)
trajectory = Trajectory(transitions=[transition])
```

## 2. Supervised Fine-Tuning (SFT) Protocol

SFT training is optimized using an **Asynchronous Look-Ahead Loop** to maximize server throughput and reduce client idling.

### The Look-Ahead Technique

This pattern enqueues the **next** batch's `forward_backward` and `optim_step` calls *before* awaiting the results of the **current** batch. This allows the server to start processing the next batch while the client is still logging metrics or preparing data for the next iteration.

### Code Template

```python
from more_itertools import chunked

batches = list(chunked(datums, batch_size))

# 1. Enqueue the very first batch
fwd_bwd_future = await training_client.forward_backward_async(data=batches[0], loss_fn="cross_entropy")
optim_future = await training_client.optim_step_async(adam_params)

# 2. Loop with Look-Ahead
for epoch in range(num_epochs):
    for step, _ in enumerate(batches):
        # Enqueue NEXT step
        next_ep, next_st = get_next_idx(epoch, step)
        if next_ep is not None:
            next_fwd_bwd_future = await training_client.forward_backward_async(data=batches[next_st], loss_fn="cross_entropy")
            next_optim_future = await training_client.optim_step_async(adam_params)
        else:
            next_fwd_bwd_future = None
            next_optim_future = None

        # Await CURRENT results
        fwd_bwd_result = await fwd_bwd_future.result_async()
        await optim_future.result_async()
        
        # Logging & Handover
        log_metrics(fwd_bwd_result.metrics)
        fwd_bwd_future = next_fwd_bwd_future
        optim_future = next_optim_future
```

## 3. Best Practices

### Flexible Metric Logging
Tinker loss metrics may have different keys depending on the configuration (e.g., `loss`, `loss:sum`, `loss:mean`). Always use a fallback mechanism when logging to the console.

```python
# Extract the first available loss metric
current_loss = metrics.get("loss")
if current_loss is None:
    for k, v in metrics.items():
        if k.startswith("loss"):
            current_loss = v
            break
```

### Advantage Calculation
For GRPO, use `prepare_minibatch_simplified` with `regularization="output_token"` to compute token-level advantages. Ensure your `TrajectoryGroup` contains enough samples ($k \ge 8$) to provide a stable baseline for advantage calculation.

### Token Handling
When constructing `TokensWithLogprobs`, ensure the `tokens` list matches exactly the sequence generated by the model, as any discrepancy will invalidate the associated logprobs during training.
