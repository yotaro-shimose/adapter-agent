# 新規Rustライブラリ選定要件

研究テーマ: AI Agent が新規 (= LLMが訓練データで触れていない) ライブラリの使い方を学習する。
そのための「学習対象ライブラリ」を crates.io から選ぶ。

## 採択基準

### 1. ドメインに関する制約
- AI / LLM 系はNG (推論にお金がかかる)
- 機械学習はNG
  - 理由1: 問題作成のコスト (意味あるデータセット/構造を生成する必要がある)
  - 理由2: 正解作成のコスト (ground truth を作るのにモデル学習や別実装の参照が必要)
- ネットワーク通信が必要なものはNG (HTTP/RPC/WebSocket/blockchain p2p 等)
- 出力がバイナリメディア (画像/音声/動画) のものはNG
  - 理由: LLM-as-a-judge が機能しない
  - 例: 色管理 (出力色), 音響合成 (波形), 画像処理 (出力画像)
  - 逆に「実行して文字列/数値/構造化データを検証する」ことは前提として OK
    (RLVR の学習信号として実行結果を使う)

### 2. ライブラリの性質
- **エンドデベロッパー向け**であること (numpy, pandas, scikit-learn の Rust 版的な位置づけ)
  - `*-core`, `*-utils`, `*-sys`, `*-derive` 等の "他ライブラリの依存先" 系はNG
- **API に厚みがあること** = 研究テーマとして成立する
  - 単一マクロ (`async_trait` 的) はNG
  - 複数の概念/モジュール/関数を学ぶ必要があること
- **library API として `docs.rs` に公開されている**こと
  - CLI binary 専用 (lib.rs を持たない) はNG
- **新規性**: `created_at >= 2025-02-01`

### 3. タスク設計の制約
- 1問あたりの解答は **単一 `Cargo.toml` + 単一 `.rs` ファイル** で収まる規模
- 入力 → 出力が **文字列/値/構造化データの比較 or LLM-as-a-judge で検証可能**であること
- コードベース全体を理解する必要があるタスクは出さない
- **問題作成コストが低い**こと
  - 例: 単位変換は変換表から問題と正解を機械的に生成できる
  - NG例: クラスタリングは意味あるデータセット生成 + 正解クラスタ割当の用意が必要

### 4. エージェント設計
- **AI Agent の行動空間**: コードを書いて実行することのみ
- **AI Agent の観測空間**: 問題文のみ (入力データ・期待出力・テストフィクスチャは見えない)
- **検証エージェントの観測空間**: 問題文 + コード + 実行結果
- 検証エージェントは「対象ライブラリを実際に使って解いているか (チートしていないか)」を判定する
- 含意:
  - 問題は対象ライブラリ特有の挙動を必要とする内容にしないと、std や別ライブラリで代替されてチートが成立する
  - APIを使わずに同等の出力を作る難易度が低いタスクは設計NG
  - 例: ECS パターンは `HashMap` で trivial に再実装可能 → チート容易
- **インターフェース構築系ライブラリはNG**: CLI/GUI/TUIアプリを生成するライブラリは、検証エージェントが構築物を実行・操作する必要があり、検証エージェントの行動空間 (コード読み + 実行結果観測) に合わない

### 4. 既に使用中のため除外
- `numrs2` (NumPy port) — 研究テーマには使うが、別軸の library を追加で選定したい
  - → 数値計算系は重複するため避ける

## 検討済み候補と判定

| ライブラリ | カテゴリ | 判定 | 理由 |
|---|---|---|---|
| moxcms | 色管理 | NG | バイナリ出力 (LLM-judge 不可) |
| naad | 音響合成 | NG | バイナリ出力 (LLM-judge 不可) |
| clump | クラスタリング | NG | 機械学習 (問題/正解の生成コスト高) |
| ez-ffmpeg | FFmpeg wrapper | NG | LLMが既知の領域 |
| fluent-i18n | i18n | NG | API薄い (実体は Project Fluent DSL) |
| rumdl | Markdownリンター | NG | CLI専用、library API 無し |
| octofhir-ucum | UCUM単位変換 | 検討中 | API厚み確認待ち |
